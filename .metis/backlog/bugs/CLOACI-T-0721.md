# trigger_rules silently ignored for packaged workflows (conditional execution / skip broken on cdylib + fleet path)

## Objective

Make per-task `trigger_rules` (conditional execution) actually work for
**packaged** workflows (cdylib-loaded, incl. the execution-agent fleet), so a
task whose trigger rule isn't satisfied is **Skipped** rather than run
unconditionally.

## Type / Priority
- [x] Bug — production issue
- [x] P1 — silently wrong: conditional execution is a documented feature that
  does nothing for packaged/fleet deployments (the standard production model).

## Impact
Any packaged workflow using `trigger_rules` — branch-not-taken, reactor
predicates, run-on-failure/skip, etc. Every gated task runs unconditionally;
nothing is ever Skipped. Discovered building a `demo_branch_workflow` to exercise
the Skipped UI state ([[CLOACI-T-0719]]): `branch_a` (run if flag==true) AND
`branch_b` (run if flag==false) — opposite conditions — both ran.

## Root Cause
`trigger_rules` were generated by the `#[task]` macro as a `trigger_rules()`
method on the task, but were **never carried across the cdylib FFI boundary**:
- The FFI task metadata (`cloacina-workflow-plugin::TaskMetadataEntry`, and the
  loader's `OwnedTaskMetadata`) had no `trigger_rules` field.
- `DynamicLibraryTask` (the host wrapper around a cdylib task) did **not**
  override `Task::trigger_rules()`, so it returned the trait default
  `{"type":"Always"}`.
- The scheduler reads `task.trigger_rules()` (`execution_planner/mod.rs:618`) →
  got `Always` for every packaged task.

(In-process / non-packaged workflows were unaffected — they call the real task's
`trigger_rules()` directly.)

## Fix
Thread the rules through the FFI metadata:
1. `cloacina-workflow-plugin/src/types.rs` — `TaskMetadataEntry` gains
   `trigger_rules: String` (`#[serde(default)]` → `Always` for old packages).
2. `cloacina-workflow-plugin/src/lib.rs` — the `package!()` `get_task_metadata`
   shell already constructs each task; now reads
   `Task::trigger_rules(&*task).to_string()` into the entry.
3. `task_registrar/types.rs` — `OwnedTaskMetadata` gains `trigger_rules_json`.
4. `task_registrar/extraction.rs` — copies it from the FFI entry.
5. `task_registrar/mod.rs` — parses it and passes it to `DynamicLibraryTask::new`.
6. `dynamic_task.rs` — stores it and overrides `fn trigger_rules()`.

Because this changes the cdylib↔host metadata contract, all fixtures must be
recompiled (done via `down -v` + rebuild) so every cdylib carries the field.

## Acceptance Criteria
- [x] A packaged workflow with a `trigger_rules`-gated task whose condition is
      unmet marks that task **Skipped** (not run). ✅
- [x] A gated task whose condition IS met runs normally. ✅
- [x] Verified end-to-end on the fleet demo. ✅
