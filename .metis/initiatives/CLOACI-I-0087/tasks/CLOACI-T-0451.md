# Protect tenant credentials in API responses and WebSocket auth (SEC-08, SEC-05)

*This template includes sections for various types of tasks. Delete sections that don't apply to your specific use case.*

## Parent Initiative **[CONDITIONAL: Assigned Task]**

[[CLOACI-I-0087]]

## Objective

Two credential exposure surfaces: (1) `POST /tenants` response includes the tenant database password and full connection string in plaintext (SEC-08), and (2) WebSocket auth tokens accepted via query parameter are logged by web servers, stored in browser history, and visible in Referer headers (SEC-05).

**Effort**: 3-4 hours

## Acceptance Criteria

## Acceptance Criteria

## Acceptance Criteria

- [ ] `POST /tenants` response no longer includes `password` or `connection_string` fields containing cleartext passwords
- [ ] Tenant credentials returned only once, clearly marked as one-time-view (or password omitted entirely, stored server-side)
- [ ] WebSocket ticket exchange endpoint: `POST /v1/auth/ws-ticket` returns `{"ticket": "<random>", "expires_at": "<timestamp>"}`
- [ ] WebSocket handler accepts ticket via query parameter as an alternative to Bearer token
- [ ] Tickets are single-use and time-limited (e.g., 60s TTL)
- [ ] Existing WebSocket auth tests updated
- [ ] Auth integration tests verify tenant response no longer leaks password

## Implementation Notes

### Technical Approach

**SEC-08 (tenant credentials)**:
In `create_tenant` response (`tenants.rs:66-78`), remove `password` and `connection_string` from the JSON response. Return only `schema_name` and `username`. The password was generated by `DatabaseAdmin::create_tenant()` — if the caller needs it, consider an alternative like returning it encrypted or requiring the caller to supply their own password.

**SEC-05 (WebSocket ticket exchange)**:
1. Add `POST /v1/auth/ws-ticket` endpoint that validates the Bearer token, generates a random ticket (UUID), stores it in a server-side `DashMap<String, (AuthenticatedKey, Instant)>` with TTL
2. WebSocket handler checks for `ticket` query param first, validates against the store, consumes (removes) on use
3. Existing header-based auth continues to work unchanged

### Dependencies
Independent of other I-0087 tasks. Can run in parallel with T-0450.
